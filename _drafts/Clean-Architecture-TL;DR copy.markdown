I am currently reading [Clean Architecture](https://www.amazon.de/Clean-Architecture-Craftsmans-Software-Structure/dp/0134494164) by Robert C. Martin and I really enjoy it, but it's quite dense in information. Thus, here I want to note and summarize some of the ideas that are mentioned in the book.

## Programming Paradigms
The three different paradigms
* Structured Programming
* Object Oriented Programming
* Functional Programming
each restrict the programmer in some way and thus impose discipline in some way. E.g. not being allowed to use <code>goto</code> statements (Structured Programming) is actually taking away a tool from the developer but results in better code on an algorithmic level.

Talking about Object Oriented Programming, the question arises, what that actually is. The three concepts **Encapsulation, Inheritance and Polymorphism** are drilled into the mind of the software developer so that they seem to be a big part of the answer. By providing different examples, Robert Martin shows that these concepts were available and used well before the advent of Object Oriented Programming. While the examples are interesting, they don't really convince me that three concepts don't make up a huge part of what Object Oriented Programming is about. Just because the concepts may be leveraged in non-object oriented languages like C does not invalidate their importance for OOP.
Anyway, it is argued that, from an architectural point of view, the most important concept of OOP is polymorphism since it enables [dependency inversion](https://en.wikipedia.org/wiki/Dependency_inversion_principle). Through that concept the flow of control in your application and the actual source code dependencies do not have to coincide any more. High level concerns like business rules of the domain don't need to depend on low level concerns like database interaction or view logic - the architect can decide on the direction of dependencies.

## SOLID
As building blocks for modules and classes in modules the SOLID-principles are revisited.

### Single Responsibility
As opposed to talking about single responsibility for functions, meaning that they should  do one thing only and be as small as possible, single responsibility for modules (or classes) means that *they should only have one reason to change*. As reasons for change usually come from changed or additional requirements, it makes sense to rephrase that into *a module (or class) should only be responsible for fulfilment of requirements from a single actor*.

I think the example of breaking this law given in the book is really interesting and is something I've also wondered about before. It's along the lines of "how much functionality should a dataclass provide besides plain fields or properties?". To make this more concrete

### Open-Closed Principle
From an architectural stand-point, the open-closed principle means that high-level concerns (like modules containing business logic) should not depend on low-level concerns (like modules dealing with presentation or database access), but dependencies should be the other way round. In that way, the code base will be open for extension (e.g. by adding another presentation technology) and the high-level concerns can be properly encapsulated from change.

## Inline HTML elements

HTML defines a long list of available inline tags, a complete list of which can be found on the [Mozilla Developer Network](https://developer.mozilla.org/en-US/docs/Web/HTML/Element).

- **To bold text**, use `<strong>`.
- *To italicize text*, use `<em>`.
- <mark>To highlight</mark>, use `<mark>`.
- Abbreviations, like <abbr title="HyperText Markup Langage">HTML</abbr> should use `<abbr>`, with an optional `title` attribute for the full phrase.
- Citations, like <cite>&mdash; Mark Otto</cite>, should use `<cite>`.
- <del>Deleted</del> text should use `<del>` and <ins>inserted</ins> text should use `<ins>`.
- Superscript <sup>text</sup> uses `<sup>` and subscript <sub>text</sub> uses `<sub>`.

Most of these elements are styled by browsers with few modifications on our part.
